# 第一章 文化根基与空间架构

## 1.1 核心文化元素：从符号到系统功能映射

玉玺、圣旨、竹箴、令牌不再是视觉装饰，而是直接对应系统的**权限层、消息层、规则层、凭证层**四大核心能力，每一项都有明确的技术落点与历史依据。

| 文化元素 | 系统定位 | UI形态 | 所属节点 | 核心作用 |
|----------|---------|--------|----------|----------|
| **传国玉玺** | 最高权限凭证 | 盘龙钮玉玺印章，钤印后指令生效 | 皇帝·天堂节点 | 最终决策签章、指令合法性校验、全局权限兜底 |
| **圣旨卷轴** | 消息协议可视化载体 | 绢本卷轴，随流转阶段加盖不同官印 | 全链路流转 | 承载 `edict/memorial/review` 全类型消息，状态可视化 |
| **竹箴典章** | 规则与知识库 | 竹简成册，收录典章制度、规范条文 | 门下省·礼部 | 审议依据、合规标准、Agent行为准则、历史案例库 |
| **鱼符令牌** | 执行权限凭证 | 铜质鱼符，六部各有专属纹样 | 尚书省派发·六部持有 | 任务执行授权、身份校验、资源配额、任务回收凭证 |

### 细节设计规则

1. **玉玺**：仅皇帝节点可调用，圣旨末尾加盖玉玺方为"中旨生效"；驳回、封驳的指令无玉玺印，不具备执行效力。
2. **圣旨**：分四阶形态——白麻（初稿）→ 黄纸（中书草拟）→ 绯绫（门下审议通过）→ 金泥（玉玺钤印·正式生效），颜色随流转阶段自动变化。
3. **竹箴**：对应"律令格式"体系，门下省审议时可一键调取对应典章，作为封驳/准奏的依据；礼部负责更新维护竹箴内容。
4. **令牌**：尚书省派发任务时同步下发对应部门令牌，任务办结后令牌缴回；令牌数量对应六部并发任务上限，超额则自动排队。

## 1.2 玉玺：系统最高权限体系（权力之源）

### 历史原型

秦汉至隋唐形成"天子六玺"制度，不同玉玺对应不同政务场景，并非一枚玉玺管所有事务；其中"传国玉玺"为正统象征，不常用，其余五枚对应日常政务。

| 玉玺名称 | 权限等级 | 对应系统功能 | 触发节点 |
|----------|---------|-------------|----------|
| **传国玉玺** | 最高级 | 全局最终决策签章、系统级配置修改、Agent生命周期管理、紧急熔断重置。**规则：加盖此玺的指令不可驳回、不可撤销，直接生效。** | 皇帝·天堂节点 |
| **皇帝行玺** | 政务级 | 日常行政任务生效签章、普通审批流程最终确认，绝大多数正式圣旨加盖此玺。 | 皇帝·日常决策 |
| **皇帝信玺** | 军事级 | 紧急任务、高优先级Bug修复、安全攻防类指令的授权签章，可跳过部分常规审批。 | 兵部/刑部 紧急任务 |
| **皇帝之玺** | 人事级 | Agent注册、权限变更、考核奖惩、岗位调动指令的专用签章。 | 吏部·人事任免 |
| **天子之玺** | 对外级 | 第三方系统对接、外部API调用、跨域资源申请的授权签章。 | 工部/对外联通 |

**UI规则**：

- 玉玺为立体朱红印章效果，印泥肌理叠加正片叠底（`--texture-yinni`）
- 未盖玺的圣旨显示"未生效"水印，六部可拒绝执行
- 封驳回退的圣旨，玉玺印自动变淡失效，重新审批后再次生效

## 1.4 竹箴典章：规则与知识库

对应隋唐"律令格式"四元法律体系，为门下省审议唯一依据：

| 分类 | 对应规则 | 示例 |
|------|---------|------|
| **律** | 系统红线规则、安全禁令、硬性约束 | 不可绕过审批、不可越权操作 |
| **令** | 部门职能规范、权责边界、流程标准 | 中书不得越权审议、兵部不得调用户部数据 |
| **格** | 奖惩细则、考核标准、权限配额 | 连续3次封驳该部门扣分、令牌上限=部门等级 |
| **式** | 操作规范、输出格式、接口标准 | API返回格式、日志记录规范 |

**UI规则**：门下省审议面板右侧常驻"竹箴侧边栏"，点击自动匹配当前任务对应条文，违规项高亮。

## 1.5 鱼符令牌：执行权限与资源凭证

| 令牌 | 对应功能 | 持有部门 |
|------|---------|----------|
| 户部鱼符 | 数据调用权限、算力配额、存储资源 | 户部 |
| 礼部鱼符 | 规范发布、文档生成、知识库编辑 | 礼部 |
| 兵部鱼符 | 代码仓库读写、部署权限、Bug修复 | 兵部 |
| 刑部鱼符 | 安全扫描、审计权限、风险处置 | 刑部 |
| 工部鱼符 | CI/CD流水线、服务器操作、工具链配置 | 工部 |
| 吏部鱼符 | Agent管理、权限分配、考核奖惩 | 吏部 |

**核心规则**：

- 尚书省派发任务时同步下发鱼符，办结后缴回注销
- 每个部门令牌数量 = 并发任务上限，用尽自动排队
- 令牌可跨部门借用，需尚书省审批，逾期自动预警

## 2.4 驾六协同模型

#### 六阶段勒令流转 (驾六图)

```
皇帝下旨 (执缰)
  │
太子承启 (理辔)
  │
┌─────────────────────────────────────────┐
│           三省审议 (校车)                │
│  中书草拟 → 门下审议 → 尚书派发          │
└─────────────────────────────────────────┘
  │         │         │         │         │         │
  ▼         ▼         ▼         ▼         ▼         ▼
户部核算   兵部研发   礼部规范   刑部审查   工部部署   吏部考核
(左服)    (右服)   (左骖)   (右骖)   (右騑)   (左騑)
  │         │         │         │         │         │
  └─────────┴─────────┴─────────┴─────────┴─────────┘
                          │
                     尚书汇总 (合辔)
                          │
                     早朝回奏 (归厩)
```

#### 六马同步与异步

```
同步 (齐头并进):
  六部同时收到勒令 → 各有专属领域 → 并行执行

异步 (前后接应):
  户部核算数据 → 兵部基于数据进行开发
  礼部输出规范 → 工部按规范部署
  刑部审计完毕 → 吏部记录考核

阻塞恢复:
  任一马失蹄 → 天子勒缰 → 尚书重新调度
  对应: task blocked → emperor notified → shangshu reassigns
```

## 3.2 封驳回流机制

### 门下省封驳条件

| 条件类型 | 说明 | 处理方式 |
|---------|------|---------|
| **合规性违规** | 触犯"律"类红线规则 | 强制封驳，附违规条款引用 |
| **权责越界** | 超出当前角色/部门权限范围 | 封驳建议调整权限或转交 |
| **资源不足** | 所需政令值/令牌超过可用量 | 封驳建议等待资源恢复或降级 |
| **质量不达标** | 方案完整性/可行性评分低于阈值 | 封驳附改进意见，返回中书重拟 |

### 封驳逆向闭环

```
[第4阶] 门下审议 → 封驳判定
  │
  ├── 生成红色圣旨 + 驳议标签
  │   内容: "臣丹心谨奏：该方案{reason}，依{lawRef}予以封驳"
  │
  └── 流转方向: 第4阶 → 第3阶 (中书重新草拟)
       │
       ├── 中书收到封驳反馈
       │   根据驳议意见修改方案
       │
       └── 重新提交第4阶审议 (最多3轮封驳循环)
            │
            └── 3轮仍未通过 → 上报天子仲裁
                 │
                 └── 天子御批: 强制通过 / 终止 / 改派
```

### 密奏直达通道

当检测到以下情况时，跳过正常审批流程，直接送达天子：

| 触发条件 | 类型 | 处理优先级 |
|---------|------|-----------|
| 安全漏洞/攻击 | P0-紧急 | 立即阻断 + 通知天子 |
| 系统崩溃风险 | P0-紧急 | 立即熔断 + 天子接管 |
| 数据泄露预警 | P0-紧急 | 立即隔离 + 天子裁决 |
| Agent异常行为 | P1-高 | 暂停该Agent + 上报 |
| 性能严重退化 | P1-高 | 自动降级 + 上报建议 |

密奏格式：
```
🔒 加密封事 (memorial_secret)

发自: {department} · {agentName}
时间: {timestamp}
密级: 绝密

事由: {incidentDescription}
证据: {evidenceLinks}
影响范围: {impactScope}
建议处置: {recommendedAction}

─── 钤印: {departmentSeal} ───
```

  id: string;
  name: string;

  department?: string;
  type: 'active' | 'passive';
  effect: number;  // multiplier e.g. 1.15 = +15%
  cost: {
    decreePoints: number;
    cooldownHours: number;
  };

}
