## 9.1 错误恢复与容错机制

### 容错设计原则

1. **优雅降级**: 功能失败时提供备选方案，而非完全不可用
2. **自动重试**: 网络错误等临时性故障自动重试（指数退避）
3. **用户提示**: 关键操作失败明确告知用户，并提供恢复入口
4. **日志记录**: 所有错误记录完整上下文，便于排查

### 错误分类与处理策略

| 错误类型 | 示例 | 处理策略 | 用户提示 | 是否阻塞 |
|---------|------|---------|---------|---------|
| **网络错误** | API请求超时 | 自动重试3次（指数退避） | "网络不稳定，正在重试..." | 否 |
| **验证错误** | 表单校验失败 | 高亮错误字段，阻止提交 | 具体字段错误提示 | 是（需用户修正） |
| **权限错误** | 无权执行操作 | 引导到权限申请页 | "您暂无此操作权限" | 是 |
| **并发冲突** | 同时编辑同一资源 | 乐观锁+合并策略 | "内容已更新，请确认" | 需用户确认 |
| **系统异常** | 未知运行时错误 | 记录日志，回滚状态 | "操作失败，已自动恢复" | 否 |
| **资源不足** | 政令值/槽位不足 | 显示资源缺口，引导获取 | "政令值不足，还需XX点" | 是 |

### 自动重试机制（指数退避）

```typescript
async function retryWithBackoff<T>(
  fn: () => Promise<T>,
  options: {
    maxRetries?: number;      // 最大重试次数（默认3）
    baseDelay?: number;       // 基础延迟ms（默认1000）
    maxDelay?: number;        // 最大延迟ms（默认10000）
  } = {}
): Promise<T> {
  const { maxRetries = 3, baseDelay = 1000, maxDelay = 10000 } = options;
  
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await fn();
    } catch (error) {
      if (attempt === maxRetries) throw error;
      
      // 指数退避 + 随机抖动
      const delay = Math.min(
        baseDelay * Math.pow(2, attempt) + Math.random() * 1000,
        maxDelay
      );
      
      await new Promise(resolve => setTimeout(resolve, delay));
    }
  }
  
  throw new Error('Max retries exceeded');
}
```

## 9.2 非阻塞交互模式规范

### UI响应性保障

| 操作类型 | 响应时间目标 | 用户体验要求 | 技术实现 |
|---------|------------|-------------|---------|
| 即时反馈 | < 100ms | 视觉立即响应 | CSS动画、本地状态更新 |
| 快速操作 | < 1s | 进度指示可选 | Loading骨架屏、禁用按钮防重复 |
| 中等耗时 | 1-3s | 必须显示进度 | Progress Bar + 百分比 + 预估时间 |
| 长时间任务 | > 3s | 可取消 + 后台执行 | 取消按钮 + 后台任务队列 |

### 异步任务队列设计

```typescript
class BackgroundTaskQueue {
  private queue: Task[] = [];
  private running = new Set<string>();
  private maxConcurrent = 3;  // 最大并发数
  
  // 入队（非阻塞）
  enqueue(task: Omit<Task, 'id'>): string {
    const taskId = generateId();
    this.queue.push({ ...task, id: taskId, status: 'queued' });
    this.processQueue();
    return taskId;
  }
  
  // 取消任务
  cancel(taskId: string): boolean { /* ... */ }
  
  // 异步处理（带错误隔离）
  private async processQueue(): Promise<void> {
    while (this.running.size < this.maxConcurrent && this.queue.length > 0) {
      const task = this.queue.shift()!;
      this.running.add(task.id);
      
      try {
        await task.execute();
        task.status = 'completed';
      } catch (error) {
        task.status = 'error';  // 单个失败不影响其他任务
      } finally {
        this.running.delete(task.id);
        this.processQueue();
      }
    }
  }
}
```

## 9.3 性能优化策略

### 渲染优化

| 优化项 | 技术 | 应用场景 | 性能提升 |
|--------|------|---------|----------|
| 虚拟列表 | `react-virtualized` | P3时间轴(13朝代)、P5旨意板(大量敕令) | 渲染1000条 vs 50条DOM |
| 懒加载 | `React.lazy` + `Suspense` | P4勋章墙、P8监控面板 | 首屏加载-40% |
| Memo缓存 | `React.memo` | 技能卡片、敕令卡片、朝臣头像 | 不相关props跳过重渲染 |
| 防抖节流 | `lodash/debounce` | 搜索框、窗口resize、滚动事件 | 事件频率降低90% |
| 离屏预计算 | Web Worker | 协同矩阵计算、勋章进度统计 | 主线程解放 |

### 数据请求优化

| 策略 | 实现 | 场景 |
|------|------|------|
| 请求合并 | 批量API `/api/batch` | 页面初始化并行请求 |
| 数据缓存 | SWR / React Query | 技能列表、朝代信息、勋章数据 |
| 增量更新 | WebSocket diff推送 | 敕令状态变更、实时指标 |
| 预加载 | `<link rel="prefetch">` | 预测下一步访问页面 |
