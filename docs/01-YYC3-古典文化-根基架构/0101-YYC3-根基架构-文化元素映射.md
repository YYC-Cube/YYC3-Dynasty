# 0101-YYC3-根基架构-文化元素映射

> **版本**：v1.0.0 ｜ **日期**：2026.06.14 ｜ **来源**：Dynasty Framework 第一章 1.1
>
> 玉玺、圣旨、竹箴、令牌不再是视觉装饰，而是直接对应系统的**权限层、消息层、规则层、凭证层**四大核心能力。

---

## 1.1 四大权力载体总览

### 文化元素→系统功能映射表

| 文化元素 | 系统定位 | UI形态 | 所属节点 | 核心作用 |
|----------|---------|--------|----------|----------|
| **传国玉玺** | 最高权限凭证 | 盘龙钮玉玺印章，钤印后指令生效 | 皇帝·天堂节点 | 最终决策签章、指令合法性校验、全局权限兜底 |
| **圣旨卷轴** | 消息协议可视化载体 | 绢本卷轴，随流转阶段加盖不同官印 | 全链路流转 | 承载 `edict/memorial/review` 全类型消息，状态可视化 |
| **竹箴典章** | 规则与知识库 | 竹简成册，收录典章制度、规范条文 | 门下省·礼部 | 审议依据、合规标准、Agent行为准则、历史案例库 |
| **鱼符令牌** | 执行权限凭证 | 铜质鱼符，六部各有专属纹样 | 尚书省派发·六部持有 | 任务执行授权、身份校验、资源配额、任务回收凭证 |

### 设计哲学三角

```
        空间骨架              权力载体              流转逻辑
   洛阳·紫微城中轴线    玉玺—圣旨—竹箴—鱼符    三省分权、六部执行
        │                      │                       │
        ▼                      ▼                       ▼
   八大地标节点映射       四大底层能力绑定         七阶全链路审批
   （天堂→定鼎门）      （权限/消息/规则/凭证）   （降旨→回奏归档）
```

**核心原则**：将古文化元素从"视觉皮肤"深度融入权限、审批、执行、监控全链路，形成一套自洽的**"古代朝堂数字化运行体系"**。所有符号均有技术落点，无纯装饰元素。

---

## 1.2 细节设计规则

### 各载体设计约束

1. **玉玺**：仅皇帝节点可调用，圣旨末尾加盖玉玺方为"中旨生效"；驳回、封驳的指令无玉玺印，不具备执行效力。
2. **圣旨**：分四阶形态——白麻（初稿）→ 黄纸（中书草拟）→ 绯绫（门下审议通过）→ 金泥（玉玺钤印·正式生效），颜色随流转阶段自动变化。
3. **竹箴**：对应"律令格式"体系，门下省审议时可一键调取对应典章，作为封驳/准奏的依据；礼部负责更新维护竹箴内容。
4. **令牌**：尚书省派发任务时同步下发对应部门令牌，任务办结后令牌缴回；令牌数量对应六部并发任务上限，超额则自动排队。

### 技术落点映射

| 载体 | 前端实现 | 后端服务 | 数据存储 |
|------|---------|---------|----------|
| 玉玺 | `SealComponent` + 钤印动效 | `AuthService.verifySeal()` | `seals.collection` |
| 圣旨 | `EdictScroll` + 阶段变色 | `EdictService.stageTransition()` | `edicts.collection` |
| 竹箴 | `LawSidebar` + 条文高亮 | `RuleEngine.matchArticles()` | `laws.subcollection` |
| 鱼符 | `TokenBadge` + 派发动画 | `ResourceService.allocateToken()` | `tokens.subcollection` |
